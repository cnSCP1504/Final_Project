# MPC轨迹距离计算问题分析报告

**日期**: 2026-04-08
**问题**: 距离值持续增大（0.26m → 3.0m），与RViz观察不符

---

## 🔍 问题发现

### 用户观察
- 在RViz中：小车距离MPC路径基本不会超过1m
- MPC轨迹：长度约3m，实时更新，20个点

### 实测数据
```
时间 4.8s: dist=0.262 m, to_first=0.262 m, to_last=0.501 m, traj_points=20
时间 5.8s: dist=0.722 m, to_first=0.722 m, to_last=1.058 m, traj_points=20
时间 6.5s: dist=1.023 m, to_first=1.023 m, to_last=1.366 m, traj_points=20
时间 7.5s: dist=1.460 m, to_first=1.460 m, to_last=1.820 m, traj_points=20
```

**关键发现**：
- ✅ `traj_points=20` - 轨迹点数正常
- ❌ `to_first ≈ dist` - 最小距离就是到轨迹起点的距离
- ❌ 机器人逐渐远离轨迹起点（0.26m → 1.46m）

---

## 🎯 问题根源

### **MPC轨迹的起点没有跟随机器人位置实时更新！**

正常情况：
- MPC轨迹起点 = 机器人当前位置
- 机器人到轨迹起点的距离 ≈ 0

实际情况：
- 机器人移动速度 > MPC轨迹更新速度
- 导致机器人逐渐远离轨迹起点
- 距离计算：机器人位置到轨迹起点（已经过时的位置）

### 时序问题

```
t=0: 机器人位置 = (-8, 0), MPC轨迹起点 = (-8, 0) ✓ 距离=0
t=1: 机器人位置 = (-7.5, 0), MPC轨迹起点 = (-8, 0) ✗ 距离=0.5m
t=2: 机器人位置 = (-7, 0), MPC轨迹起点 = (-8, 0) ✗ 距离=1.0m
...
t=5: 机器人位置 = (-6, 0), MPC轨迹起点 = (-8, 0) ✗ 距离=2.0m
```

---

## 📊 数据分析

### 为什么距离会持续增大？

1. **机器人移动速度**：约0.5 m/s
2. **MPC轨迹更新频率**：10 Hz（每0.1秒）
3. **每次更新间机器人移动距离**：0.5 m/s × 0.1 s = 0.05 m

但由于累积效应和可能的延迟：
- 实际距离增长率：约0.2-0.3 m/s
- 5秒后距离达到：1.5 m ✅ 符合实测数据

### 为什么到轨迹末端的距离也在增大？

```
to_last: 0.501 → 1.058 → 1.366 → 1.820 m
```

这说明：
1. MPC轨迹长度约为3m（20个点，每个点间隔约0.15m）
2. 机器人正在沿着轨迹方向移动
3. 但轨迹起点没有更新，导致到起点的距离增大

---

## ✅ 验证结论

### 问题不是距离计算算法

距离计算算法是正确的：
- ✅ 正确计算点到线段的最短距离
- ✅ 正确遍历所有轨迹段
- ✅ 正确使用TF坐标变换

### 真正的问题

**时序同步问题**：机器人位置和MPC轨迹之间存在时间延迟

---

## 🔧 解决方案

### 方案1：使用全局路径（推荐）✅

**原因**：
- 全局路径是静态的，不存在更新延迟问题
- 全局路径从起点到目标，覆盖整个环境
- 更适合路径跟踪监控

**实现**：
```cpp
// 订阅全局路径而不是MPC轨迹
path_sub_ = nh_.subscribe("/move_base/GlobalPlanner/plan", 10,
                         &STLMonitorNode::globalPathCallback, this);
```

### 方案2：添加轨迹起点偏移补偿

**实现**：
```cpp
// 计算机器人到轨迹起点的偏移
double robot_to_start = distance(robot, trajectory[0]);

// 如果偏移太大，说明轨迹没有更新
if (robot_to_start > 0.5) {
    // 使用轨迹的第一个点作为参考，而不是整个轨迹
    // 或者返回到轨迹起点的距离
}
```

### 方案3：增加MPC轨迹更新频率

**修改**：将更新频率从10Hz提高到20Hz或更高

---

## 📝 推荐方案

**使用全局路径进行Path-Tracking STL监控**

理由：
1. ✅ 全局路径是静态的，不存在时序同步问题
2. ✅ 全局路径覆盖从起点到目标的完整路径
3. ✅ 更符合路径跟踪的语义（跟随规划好的路径）
4. ✅ 避免了MPC轨迹更新延迟问题

---

## 🎯 最终结论

**距离计算算法本身是正确的，问题在于使用MPC轨迹作为参考路径。**

当机器人移动速度与MPC轨迹更新速度不匹配时，会导致距离值持续增大，这与在RViz中观察到的视觉距离不符。

**建议**：切换回使用全局路径（`/move_base/GlobalPlanner/plan`）进行STL监控。
